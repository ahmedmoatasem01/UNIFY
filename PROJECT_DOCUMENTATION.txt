3.1 ARCHITECTURE OVERVIEW
--------------------------

Unify follows a layered architecture pattern with clear separation of concerns:

┌─────────────────────────────────────────────────────────────┐
│                    PRESENTATION LAYER                        │
│  Templates (HTML) + Static Files (CSS/JS) + Frontend Logic  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    CONTROLLER LAYER                          │
│  Flask Blueprints - HTTP Request/Response Handling          │
│  (auth_controller, task_controller, ai_assistant_controller)│
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    SERVICE LAYER                             │
│  Business Logic - RAG Engine, Course Optimization, etc.    │
│  (ai_assistant_service, course_optimization_service)        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    REPOSITORY LAYER                         │
│  Data Access - Database Queries (via RepositoryFactory)      │
│  (user.repository, task.repository, knowledge_base.repository)│
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    DATA LAYER                               │
│  SQL Server Database + Models (Data Structures)            │
│  (User, Student, Task, Course, Enrollment, etc.)           │
└─────────────────────────────────────────────────────────────┘

Key Components:
- Application Factory Pattern: create_app() function
- Blueprint Pattern: Modular route organization
- Repository Pattern: Data access abstraction
- Service Layer: Business logic separation
- Singleton Pattern: Database connection management

3.2 MVC STRUCTURE
-----------------

Model-View-Controller (MVC) architecture implementation:

MODELS (src/models/)
  Purpose: Data structures and business entities
  Examples:
    - User: User_ID, Username, Email, Password_Hash, Created_At
    - Student: Student_ID, User_ID, Department, Year_Level, GPA
    - Task: Task_ID, Student_ID, Task_Title, Due_Date, Priority, Status

VIEWS (src/templates/)
  Purpose: HTML templates for user interface
  Structure:
    - Base template with sidebar navigation
    - Feature-specific templates (overview.html, tasks.html, etc.)
    - Component templates (topbar_right.html)
    - AI Assistant interface (ai_assistant.html)

CONTROLLERS (src/controllers/)
  Purpose: HTTP request handlers, route definitions
  Blueprints:
    - auth_controller.py: Authentication routes
    - user_controller.py: User management
    - student_controller.py: Student operations
    - task_controller.py: Task CRUD operations
    - course_controller.py: Course management
    - enrollment_controller.py: Enrollment operations
    - schedule_controller.py: Schedule management
    - calendar_controller.py: Calendar events
    - course_registration_controller.py: Course registration

Data Flow:
  Request → Controller → Service → Repository → Database
  Database → Repository → Service → Controller → Response (JSON/HTML)

3.3 API ENDPOINTS
-----------------

A. Authentication Endpoints (/auth)
   POST   /auth/login
          - Login user
          - Body: {email, password}
          - Returns: {message, user} or {error}

   POST   /auth/register
          - Register new user
          - Body: {username, email, password}
          - Returns: {message, user} or {error}

   POST   /auth/logout
          - Logout user
          - Returns: {message}

   GET    /auth/me
          - Get current user info
          - Returns: {user} or {error}

B. User Endpoints (/users)
   GET    /users/
          - Get all users
          - Returns: List of users

   GET    /users/<user_id>
          - Get user by ID
          - Returns: User object or 404

   POST   /users/create
          - Create new user
          - Body: {username, email, password}
          - Returns: {message, user}

C. Task Endpoints (/tasks)
   GET    /tasks/api
          - Get all tasks
          - Returns: List of tasks

   GET    /tasks/api/<task_id>
          - Get task by ID
          - Returns: Task object

   GET    /tasks/api/student/<student_id>
          - Get tasks for student
          - Returns: List of tasks

   GET    /tasks/api/user
          - Get tasks for current user
          - Returns: List of tasks

   POST   /tasks/api
          - Create new task
          - Body: {Task_Title, Due_Date, Priority, Status}
          - Returns: {message, task}

   PUT    /tasks/api/<task_id>
          - Update task
          - Body: {Task_Title, Due_Date, Priority, Status}
          - Returns: {message, task}

   DELETE /tasks/api/<task_id>
          - Delete task
          - Returns: {message}

D. AI Assistant Endpoints (/ai-assistant)
   GET    /ai-assistant/
          - AI Assistant main page
          - Returns: HTML page

   POST   /ai-assistant/ask
          - Ask question to AI Assistant
          - Body: {question, model: 'unify'|'ollama'}
          - Returns: {success, answer, sources, confidence}

   GET    /ai-assistant/history
          - Get chat history
          - Query: ?limit=50
          - Returns: {success, history}

   POST   /ai-assistant/clear-history
          - Clear chat history
          - Returns: {success, message}

   GET    /ai-assistant/categories
          - Get knowledge base categories
          - Returns: {success, categories}

   GET    /ai-assistant/knowledge-base
          - Get knowledge base documents
          - Query: ?category=CategoryName
          - Returns: {success, documents}

E. Course Endpoints (/courses)
   GET    /courses/
          - Get all courses
          - Returns: List of courses

   GET    /courses/<course_id>
          - Get course by ID
          - Returns: Course object

   POST   /courses/create
          - Create new course
          - Body: {Course_Name, Credits, Instructor_ID, Schedule}
          - Returns: {message, course}

F. Enrollment Endpoints (/enrollments)
   GET    /enrollments/student/<student_id>
          - Get enrollments for student
          - Returns: List of enrollments

   POST   /enrollments/enroll
          - Enroll student in course
          - Body: {Student_ID, Course_ID}
          - Returns: {message, enrollment}

G. Schedule Endpoints (/schedule)
   GET    /schedule/
          - Get schedule page
          - Returns: HTML page

   GET    /schedule/api
          - Get schedule data
          - Returns: Schedule information

H. Calendar Endpoints (/calendar)
   GET    /calendar/
          - Get calendar page
          - Returns: HTML page

   GET    /calendar/api
          - Get calendar events
          - Returns: List of events

   POST   /calendar/api
          - Create calendar event
          - Body: {Title, Date, Time, Source}
          - Returns: {message, event}

I. Course Registration Endpoints (/course-registration)
   GET    /course-registration/
          - Course registration page
          - Returns: HTML page

   POST   /course-registration/register
          - Register for course
          - Body: {Course_ID}
          - Returns: {message, enrollment}

J. Transcript Endpoints (/transcript)
   GET    /transcript/
          - Transcript page
          - Returns: HTML page

   GET    /transcript/api
          - Get transcript data
          - Returns: {courses, gpa, total_credits}

K. Message Endpoints (/messages)
   GET    /messages/
          - Messages page
          - Returns: HTML page

   GET    /messages/api
          - Get messages
          - Returns: List of messages

   POST   /messages/api
          - Send message
          - Body: {Receiver_ID, Message_Text}
          - Returns: {message, message_obj}

L. Overview Endpoints (/overview)
   GET    /overview/
          - Overview dashboard
          - Returns: HTML page

   GET    /overview/api/stats
          - Get dashboard statistics
          - Returns: {courses_count, tasks_count, events_count}

3.4 DATABASE SCHEMA
-------------------

Database: unify
Database System: SQL Server (also supports MySQL)

Main Tables:

1. User
   - User_ID (INT, PRIMARY KEY, IDENTITY)
   - Username (NVARCHAR(100), UNIQUE, NOT NULL)
   - Email (NVARCHAR(255), UNIQUE, NOT NULL)
   - Password_Hash (NVARCHAR(255), NOT NULL)
   - Created_At (DATETIME, DEFAULT GETDATE())

2. Student
   - Student_ID (INT, PRIMARY KEY, IDENTITY)
   - User_ID (INT, FOREIGN KEY → User.User_ID)
   - Department (NVARCHAR(100))
   - Year_Level (INT, CHECK 1-5)
   - GPA (FLOAT, CHECK 0.0-4.0)

3. Instructor
   - Instructor_ID (INT, PRIMARY KEY, IDENTITY)
   - User_ID (INT, FOREIGN KEY → User.User_ID)
   - Department (NVARCHAR(100))
   - Office (NVARCHAR(100))
   - Email (NVARCHAR(255), UNIQUE)

4. Course
   - Course_ID (INT, PRIMARY KEY, IDENTITY)
   - Course_Name (NVARCHAR(200), NOT NULL)
   - Credits (INT, NOT NULL)
   - Instructor_ID (INT, FOREIGN KEY → Instructor.Instructor_ID)
   - Schedule (NVARCHAR(MAX))

5. Enrollment
   - Enrollment_ID (INT, PRIMARY KEY, IDENTITY)
   - Student_ID (INT, FOREIGN KEY → Student.Student_ID)
   - Course_ID (INT, FOREIGN KEY → Course.Course_ID)
   - Status (NVARCHAR(20), CHECK: 'enrolled', 'dropped', 'completed')
   - Grade (NVARCHAR(5))
   - Semester (NVARCHAR(50))
   - UNIQUE (Student_ID, Course_ID)

6. Task
   - Task_ID (INT, PRIMARY KEY, IDENTITY)
   - Student_ID (INT, FOREIGN KEY → Student.Student_ID)
   - Task_Title (NVARCHAR(255), NOT NULL)
   - Due_Date (DATETIME, NOT NULL)
   - Priority (NVARCHAR(10), CHECK: 'low', 'medium', 'high')
   - Status (NVARCHAR(20), CHECK: 'pending', 'completed')

7. Schedule
   - Schedule_ID (INT, PRIMARY KEY, IDENTITY)
   - Student_ID (INT, FOREIGN KEY → Student.Student_ID)
   - Course_ID (INT, FOREIGN KEY → Course.Course_ID)
   - Day_Of_Week (NVARCHAR(20))
   - Start_Time (TIME)
   - End_Time (TIME)
   - Location (NVARCHAR(200))

8. Calendar
   - Event_ID (INT, PRIMARY KEY, IDENTITY)
   - Student_ID (INT, FOREIGN KEY → Student.Student_ID)
   - Title (NVARCHAR(255), NOT NULL)
   - Date (DATE)
   - Time (TIME)
   - Source (NVARCHAR(50))

9. Message
   - Message_ID (INT, PRIMARY KEY, IDENTITY)
   - Sender_ID (INT, FOREIGN KEY → User.User_ID)
   - Receiver_ID (INT, FOREIGN KEY → User.User_ID)
   - Message_Text (NVARCHAR(MAX), NOT NULL)
   - Is_Read (BIT, DEFAULT 0)
   - Created_At (DATETIME, DEFAULT GETDATE())

10. Knowledge_Base
    - KB_ID (INT, PRIMARY KEY, IDENTITY)
    - Title (NVARCHAR(500), NOT NULL)
    - Content (NVARCHAR(MAX), NOT NULL)
    - Category (NVARCHAR(100))
    - Keywords (NVARCHAR(500))
    - Created_Date (DATETIME, DEFAULT GETDATE())
    - Updated_Date (DATETIME, DEFAULT GETDATE())
    - Source (NVARCHAR(500))

11. Chat_History
    - Chat_ID (INT, PRIMARY KEY, IDENTITY)
    - User_ID (INT, FOREIGN KEY → User.User_ID)
    - Question (NVARCHAR(MAX), NOT NULL)
    - Answer (NVARCHAR(MAX), NOT NULL)
    - Sources (NVARCHAR(MAX))
    - Created_Date (DATETIME, DEFAULT GETDATE())

12. AI_Note
    - Note_ID (INT, PRIMARY KEY, IDENTITY)
    - Student_ID (INT, FOREIGN KEY → Student.Student_ID)
    - Original_Text (NVARCHAR(MAX))
    - Summary (NVARCHAR(MAX))
    - File_Name (NVARCHAR(255))
    - File_Type (NVARCHAR(50))
    - Created_At (DATETIME, DEFAULT GETDATE())

Indexes:
- User: Email, Username
- Student: User_ID
- Enrollment: Student_ID, Course_ID
- Task: Student_ID, Due_Date, Status
- Message: Sender_ID, Receiver_ID

3.5 DESIGN PATTERNS
-------------------

1. Application Factory Pattern
   Implementation: create_app() function in src/app.py
   Purpose: Flexible application creation for testing and deployment
   Benefits: Easy configuration, testing support, multiple instances

2. Repository Pattern
   Implementation: RepositoryFactory in src/repositories/repository_factory.py
   Purpose: Abstract data access layer
   Benefits: Easy testing (mocking), database independence, clean separation

3. Singleton Pattern
   Implementation: DatabaseConnection in src/core/db_singleton.py
   Purpose: Single database connection instance
   Benefits: Resource efficiency, connection pooling, state management

4. Service Layer Pattern
   Implementation: Services in src/services/
   Purpose: Business logic separation from controllers
   Benefits: Reusability, testability, maintainability

5. Blueprint Pattern
   Implementation: Flask Blueprints in src/controllers/
   Purpose: Modular route organization
   Benefits: Code organization, scalability, team collaboration

6. MVC Pattern
   Implementation: Models, Views (Templates), Controllers
   Purpose: Separation of concerns
   Benefits: Maintainability, testability, scalability

3.6 ASSUMPTIONS AND CONSTRAINTS
--------------------------------

Assumptions:

1. Database:
   - SQL Server is available and running
   - Database 'unify' exists or will be created
   - User has appropriate database permissions
   - Trusted connection (Windows Authentication) or credentials provided

2. Environment:
   - Python 3.8+ installed
   - All dependencies installable via pip
   - Port 5000 available for Flask application
   - Windows environment (primary, but cross-platform compatible)

3. User Roles:
   - Three roles: Student, Instructor, Admin
   - Students can register for courses
   - Instructors can manage courses
   - Admins have full access

4. AI Features:
   - Ollama is optional (system works without it)
   - AI Note summarization requires transformers library
   - Hugging Face models downloadable (internet connection)

5. File Uploads:
   - Supported formats: PDF, DOCX, TXT
   - File size limits: Reasonable (not explicitly enforced)
   - Upload directory: src/uploads/

Constraints:

1. Database:
   - SQL Server syntax required (not pure MySQL)
   - Table names may need brackets: [User] instead of User
   - IDENTITY instead of AUTO_INCREMENT
   - NVARCHAR instead of TEXT

2. Performance:
   - Response time target: <2 seconds (NFR1)
   - Concurrent users: Up to 500 (NFR4)
   - Database connection pooling via singleton

3. Security:
   - Passwords must be hashed (SHA-256)
   - No plaintext password storage
   - Session-based authentication
   - SQL injection prevention via parameterized queries

4. Scalability:
   - Single database instance (not distributed)
   - File storage on local filesystem
   - No load balancing (single server)

5. Browser Compatibility:
   - Modern browsers (Chrome, Firefox, Edge, Safari)
   - JavaScript enabled required
   - CSS3 support required
